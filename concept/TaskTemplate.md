# Шаблон задачи

Имя файла: `Tasks/TASK-XXX-Короткое-Название.md` (PascalCase с дефисами).

Жизненный цикл задачи проходит **три роли** (три команды):

| Команда      | Кто                       | Что делает                                                        |
|--------------|---------------------------|-------------------------------------------------------------------|
| `/pm-task`   | Менеджер/PM               | Описывает желание: что должно работать и как для пользователя. Без кода |
| `/lead-task` | Тимлид (Sonnet/Opus)      | Анализирует код, пишет пошаговый разбор для исполнителя. Определяет сложность |
| `/run-task`  | Исполнитель (Haiku/Sonnet/Opus по сложности) | Делает строго по разбору. Не импровизирует |

## Шаблон файла задачи

```markdown
# TASK-XXX: [Название]

## Связь
- **F-XX** из Concept.md
- **D-XX** из Diff.md  (если применимо)
- **BUG-XX** из Bugs.md  (если применимо)
- **Приоритет:** P0 / P1 / P2

---

## 📋 Постановка от менеджера

_Автор: [PM]_
_Дата: YYYY-MM-DD_

### Что хотим
[2-4 предложения бизнес-описания]

### Пользовательский сценарий
1. ...
2. ...

### Acceptance criteria
- [ ] Когда X — то Y
- [ ] Когда A — то B

### Дизайн
[ссылка на DesignConcept.md → раздел; конкретные токены; макет] ИЛИ «Не применимо»

### Done-критерий
_Из Concept.md F-XX:_ [скопировать дословно]

---

## 🛠 Технический разбор от тимлида

_Автор: tech-lead_
_Дата: YYYY-MM-DD_
_Статус: [ ] нужен разбор / [x] готов_

### Анализ текущего состояния
- В коде уже есть: ...
- Что переиспользуем: ...
- Что дописать: ...

### Архитектурное решение
[1-3 абзаца: КАК делаем и ПОЧЕМУ]

### Пошаговая декомпозиция (для исполнителя)
1. **[Шаг]** — файл `путь`, метод `имя(...)`, что делает
2. ...

### Edge cases (явно обработать)
- [ ] ...

### Файлы для изменения
- `путь` — что меняется

### Файлы НЕ трогать
- `путь` — почему

### Команды проверки (для DoD)
- Компиляция: `xcodebuild -scheme CityDeveloper build`
- Тесты: `xcodebuild test -scheme CityDeveloper`
- Ручная проверка: [как именно]

### Сложность: `junior` / `middle` / `senior`
### Объём: S / M / L

---

## ✅ Исполнение

_Исполнитель: —_

### Definition of Done

#### Функциональные
- [ ] Все Acceptance criteria выполнены
- [ ] Done-критерий проверен в реальном использовании

#### Технические
- [ ] Компиляция/линтер без новых ошибок
- [ ] Тесты не сломаны
- [ ] Нет хардкод-строк (пути, имена файлов, токены — в констатах/конфигах)

#### Обновление документации (обязательно)
- [ ] `Current.md`: F-XX → ✅ (или ⚠️ с уточнением)
- [ ] `Diff.md`: D-XX удалён
- [ ] `Bugs.md`: BUG-XX закрыт (если есть)
- [ ] Новые идеи → `Backlog.md`, новые баги → `Bugs.md`

---

## Статус

`[x] waiting-for-lead` / `[ ] ready` / `[ ] in-progress` / `[ ] review` / `[ ] done`

## Метаданные
- Создана PM: YYYY-MM-DD
- Готова к работе: —
- Завершена: —
- Коммит: —
```

## Принципы

1. **Три раздела — три ответственности.** PM не лезет в код. Тимлид не пишет финальный код.
   Исполнитель не импровизирует.
2. **Один TASK = один атомарный коммит.**
3. **Задача done только при полном DoD.** Главное — обновление документации, иначе
   `diff` не пересчитается.
4. **Не расширять `Concept.md` из задачи.** Новые идеи → `Backlog.md`.
5. **Нумерация TASK-001, TASK-002…** Не переиспользовать номера.
6. **Сложность определяет модель исполнителя:** junior=Haiku, middle=Sonnet, senior=Opus.

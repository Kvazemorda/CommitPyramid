# Правила генерации спрайтов (юниты и биомы)

Документ описывает, как готовить растровые спрайты для CityDeveloper, чтобы они
ложились в изометрический рендер SpriteKit без подгонки в коде и держали единый
стиль во всём городе.

## 1. Технические параметры

Единые для **всех** AI-картинок, иначе они не сядут в общий рендер.

| Параметр              | Значение                                                |
|-----------------------|---------------------------------------------------------|
| Формат                | PNG, **8-bit RGBA** (с альфой)                          |
| Разрешение            | **1024×1024** или **1217×1217** (квадрат, ровно один объект) |
| Фон                   | **Полностью прозрачный** (никаких подложек / рамок / теней) |
| Ракурс                | Изометрия 2:1 (классический «фараоновский»), угол сверху-справа, без перспективы и без вращения камеры |
| Освещение             | Свет сверху-слева (грани слева — светлее, справа — темнее, верх — самый светлый) |
| Композиция            | Объект **по горизонтальному центру**, основание ≈ в нижней трети картинки (anchor ≈ `(0.5, 0.30)`) |
| Цветовой охват        | 24–48 цветов на спрайт, без HDR / неона                 |
| Стилистика            | Лёгкая иллюстративность, мягкий outline (1–2 px), без фотореализма и без cel-shade |

Тайл в движке — **64×32**. Спрайт по ширине рендерится примерно «в один тайл» (64 px)
или «в два тайла» (128 px) для крупных зданий. То есть детализация исходного PNG
видна только при максимальном зуме — не разменивай разрешение на мелкие надписи.

## 2. Стиль (общий для всех ассетов)

- **Палитра:** тёплые охра, глина, песок, серый камень, выцветшее дерево; синий —
  только для воды и редких акцентов (двери, окна — приглушённый сине-серый).
  Ориентир — `Sources/CityDeveloper/Theme/Palette.swift`.
- **Возрастная фактура:** материалы потёрты, штукатурка с трещинами, дерево
  выгоревшее. **Никаких** пластиковых поверхностей и блеска.
- **Outline:** аккуратный, цветной (тёмно-коричневый/угольный, не чёрный
  RGB(0,0,0)). Толщина 1–2 px, чтобы не «звенел» при `SKTexture.filteringMode = .linear`.
- **Тени и декор земли:** **не рисуем на спрайте**. Тень-эллипс и тайл-земля
  добавляются кодом в `UnitSprites.makeStageNode`. На PNG здание стоит «в пустоте».
- **Эволюция:** соседние стадии (stage N и N+1) должны читаться как «улучшение того
  же дома», а не как два независимых здания. Цветовая семья и силуэт основания —
  родственные; меняется материал, высота, детализация.

## 3. Куда складывать

В пакете SwiftPM (`Package.swift` уже подключает `resources: [.process("Resources")]`):

```
Sources/CityDeveloper/Resources/
├─ Buildings/         # сейчас — обычная папка, готова к миграции в .atlas
│  ├─ shack.png
│  ├─ house_stage2.png
│  └─ …
├─ Citizens/
└─ Terrain/
```

Когда зданий станет ≥ 4 — переименовать `Buildings/` → `Buildings.atlas/`.
SpriteKit при сборке сам упакует PNG в один атлас, код менять не придётся (загрузка
через `SKTexture(imageNamed:)` / `Bundle.module.url(forResource:withExtension:)`).

**Naming convention.** Latin snake_case, `<kind>_stage<N>.png`. Имя `kind` = `UnitKind.rawValue`
из `Sources/CityDeveloper/Data/CityState.swift`. Stage без суффикса = stage 1
(используется для категориальной лачуги/well/raw — там стадия одна).

| Файл                    | Что                                          |
|-------------------------|----------------------------------------------|
| `shack.png`             | Лачуга (residential stage 1)                 |
| `house_stage2.png`      | Деревянный дом (residential stage 2)         |
| `house_stage3.png`      | Каменный дом (residential stage 3)           |
| `house_stage4.png`      | Многоэтажный (residential stage 4)           |
| `villa.png`             | Вилла (residential stage 5)                  |
| `well.png` / `road.png` / `warehouse_stageN.png` | Инфраструктура   |
| `workshop_stageN.png` / `raw.png` | Производство                       |
| `market_stageN.png` / `forum_stageN.png` / `temple_stageN.png` / `obelisk.png` | Социальное |

## 4. Каталог зданий (что именно генерировать)

Целевой объём — **50 юнитов** по 6 категориям. Полный реестр с terrain, size,
minStage, large и эволюционными цепочками — `concept/Concept.md` § F-16
(является source of truth, дублировать сюда не надо). Здесь — только сводка по
объёму работы и сноска на текущую реализацию.

### 4.1. Итог по категориям (см. F-16)

| Категория      | Шт. | Примеры                                          |
|----------------|-----|--------------------------------------------------|
| Жилые          | 12  | Землянка, Лачуга, Хижина, Фермерский дом, Дом, Двухэтажный, Каменный, Таунхаус, Доходный, Усадьба, Вилла, Дворец |
| Инфраструктура | 8   | Колодец, Дорога, Ворота, Мост, Цистерна, Маяк, Канал, Пристань |
| Производство   | 12  | Ферма, Причал, Мастерская, Склад, Кузница, Гончарня, Пивоварня, Лесопилка, Каменоломня, Шахта, Большой склад, Завод |
| Социальное     | 10  | Таверна, Рынок, Площадь, Баня, Школа, Больница, Форум, Библиотека, Акведук, Театр |
| Религиозное    | 5   | Часовня, Храм, Обелиск, Собор, Пирамида           |
| Военное        | 3   | Сторожевая башня, Казармы, Верфь                  |

### 4.2. Эволюционные цепочки

Цепочки из F-16 — это **визуальная подмена** клеток, не переразмещение. Для них
спрайты «до» и «после» **обязаны** держать одну стилистическую семью (силуэт
основания, палитра, кровельные материалы):

- Землянка → Лачуга → Дом → Доходный дом
- Хижина → Каменный дом → Усадьба
- Двухэтажный → Доходный дом
- Фермерский → Усадьба
- Склад → Большой склад

Удобно генерить целыми цепочками в одном диалоге (см. § 6) — стиль не разваливается.

### 4.3. Текущая реализация (что уже отрисовано кодом)

В `UnitSprites.makeCategoricalBuilding` пока живёт упрощённый набор 4 категории × 5 stage
(`UnitKind` = shack/house/villa/well/road/warehouse/workshop/raw/market/forum/temple/obelisk).
Эта 12-юнитная модель — переходный слой к F-16; новые ассеты можно класть в
`Resources/Buildings/` **уже с финальными именами по F-16** (`zemlyanka.png`,
`khizhina.png`, `dvorets.png` и т.д.), даже если код пока их не подгружает —
интегрируем по мере расширения `UnitKind`.

### 4.4. Размер footprint → размер спрайта

`size` из F-16 определяет, во сколько тайлов сядет здание:

| Footprint | Целевая ширина спрайта при загрузке | Рекомендуемое разрешение PNG |
|-----------|------------------------------------|------------------------------|
| 1×1       | 64 px (1 тайл)                     | 1024² достаточно              |
| 1×2 / 2×1 | 96–112 px                          | 1024²                          |
| 2×2       | 128 px (2 тайла)                   | 1024² или 1217²                |
| 2×3 / 3×2 | 160–192 px                         | 1217² или 1536²                |
| 3×3       | 192 px                             | 1536²                          |
| 4×4       | 256 px (Пирамида и т.п.)           | 2048²                          |

Соответственно у крупных юнитов разрешение исходного PNG больше — деталь
читается даже на дальнем зуме.

### 4.5. Декор биомов (отдельный набор спрайтов)

Помимо 50 юнитов нужен **decor-набор**, который движок раскидывает на тайлах
биома (не привязан к клеткам):

| Биом     | Декор                                                         |
|----------|---------------------------------------------------------------|
| Луг      | Кусты, полевые цветы, отдельные камни, тропинки               |
| Лес      | 3–4 вида деревьев (сосна, лиственное, сухостой), пни, валуны   |
| Пустыня  | Пальмы, кактусы, скелеты, барханы-декор                       |
| Горы     | Скальные выступы, валуны, осыпь                               |
| Камни    | Россыпи камней разной формы                                   |
| Река     | Камыш, лилии, мостки                                          |
| Море     | Прибойная пена, морские камни у берега                        |

Складывать в `Resources/Terrain/decor/<biome>/`. По 3–5 вариаций на биом, чтобы
poisson-раскладка не выглядела копипастой. Это ещё ~25–35 PNG.

**Итого визуальный бэклог:** ≈ 50 зданий (часть с 1–3 эволюционными формами,
итого ~70 спрайтов зданий) + ~30 декор-спрайтов + ~21 тайл биомов (7 биомов ×
3 вариации) + ~30 переходных тайлов = **~150 PNG** на «полный» каталог.

## 5. Биомы (terrain тайлы)

См. `concept/Concept.md`, F-15. Семь биомов, нужны бесшовные изометрические
тайлы 64×32 px (для атласа землю можно генерировать в 256×128 — потом downscale).

| Биом     | Что генерировать                                    |
|----------|-----------------------------------------------------|
| Луг      | Зелёная трава с лёгкими пучками, дефолт             |
| Пустыня  | Светлый песок, мелкая рябь, редкие камешки          |
| Лес      | Тёмная трава (тайл без деревьев — деревья отдельным sprite-overlay) |
| Горы     | Скальная порода, фактура камня                      |
| Камни    | Каменистая почва, переходная зона                   |
| Река     | Голубая вода с лёгкой рябью                         |
| Море     | Сине-зелёная вода, более глубокий тон               |

Для каждого биома нужны **3 варианта**, чтобы не было «шахматного» повторения, и
**4 переходных** тайла к соседнему биому (луг↔лес, луг↔пустыня, камни↔горы,
суша↔вода). Переходы делаются альфа-маской по углам.

Декор биома (деревья, камни, кусты, оазис) — **отдельные** спрайты с прозрачным
фоном, складываются в `Resources/Terrain/decor/`. Это позволит движку
расставлять их poisson-распределением, а не пытаться угадать ритм на тайле.

## 6. Workflow генерации

**Один спрайт = один PNG.** AI-модели лучше держат детализацию и стиль на
отдельных запросах, чем на «сетке 4×5».

**Батч по 2–3 в одном диалоге.** Делаешь stage 2 → сразу проси stage 3 в том же
диалоге, с явным style-prompt:

> «Тот же стиль, та же палитра, тот же ракурс, тот же масштаб основания. Это
> следующая стадия того же жилого здания — каменный дом вместо деревянного».

Это держит силуэт и палитру согласованными, без расхождений между стадиями.

**Style-anchor.** Первый «эталонный» спрайт (сейчас — `shack.png`) держим как
референс. Любой новый запрос начинать с фразы «в стиле прикреплённого
референса» + сам файл. Если ассет сильно выпадает — перегенерировать, **не**
пытаться чинить в Photoshop.

**Проверка перед коммитом:**
1. Прозрачный фон (`file Resources/.../foo.png` → `RGBA, non-interlaced`).
2. Высота ≈ ширине (квадрат). Если нет — обрезать в квадрат, иначе anchorY поплывёт.
3. Запустить сборку, посадить юнит в игре, проверить:
   - основание не «висит» над тайлом и не утоплено;
   - силуэт не сливается с соседними юнитами;
   - читабельно на минимальном зуме (×0.5–0.7).

Если основание плывёт — править **в коде**, `anchorY` в вызове
`loadBuildingSprite(...)` (стандартный диапазон `0.25..0.35`), а не перерисовывать PNG.

## 7. Чего НЕ делать

- **Не** просить AI генерить «sprite sheet 4×5 в одном PNG» — детализация на каждый
  объект упадёт в 4–5 раз и стиль поплывёт.
- **Не** рисовать тень и землю прямо на спрайте — рендер сам кладёт shadow + ground.
- **Не** ставить рамку, белый фон, надписи, водяные знаки.
- **Не** менять ракурс между стадиями (всё в одной изометрии 2:1).
- **Не** хранить psd / source-файлы внутри `Resources/` — только финальные PNG,
  иначе SwiftPM попытается включить их в bundle.
- **Не** добавлять анимационные кадры на этом этапе. Анимация (дым из трубы,
  огоньки в окнах) — отдельная задача (overlay-ноды поверх статического спрайта).

## 8. Чек-лист на новый спрайт

1. Сгенерировать PNG (1024² или 1217², прозрачный фон, ракурс/свет по правилам).
2. Назвать по конвенции (`<kind>[_stage<N>].png`), Latin snake_case.
3. Положить в `Sources/CityDeveloper/Resources/<Buildings|Citizens|Terrain>/`.
4. Прогнать `swift build` — проверить, что `resource_bundle_accessor.swift`
   увидел файл.
5. Запустить игру, посадить юнит, проверить посадку и силуэт.
6. При необходимости — подстроить `anchorY` / `targetWidth` в коде, **не** в PNG.

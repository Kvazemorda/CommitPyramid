## Идеи

- [ ] **Жители как драйвер роста города и stage квартала** — последовательная
      механика развития, население — обязательное условие апгрейдов:
      1. Стартовый квартал ставит дома + колодец + простую инфраструктуру.
      2. Жители приходят по магистрали в дома (уже работает: см. CitizenManager.spawnCitizen
         → entryPoint магистрали; CityEngine targetCount по residentialCount).
      3. Переход на следующую stage квартала требует набора N жителей (сейчас
         StageRules смотрит только taskCount/ageDays — добавить ветку «populationGate»).
      4. На stage ≥ 2 в каталоге кварталов открываются «рабочие» петли (production,
         social) — они дают рабочие места жильцам и привлекают БОЛЬШЕ жителей.
      5. Появление нового project'а в городе требует, чтобы суммарное население
         перевалило за порог (городское «attract» условие).
      Контракт: без жителей город не растёт, жители не приходят без жилья — петля
      обратной связи. Реализация: новое поле populationRequired на UnitKind/Stage и
      проверка в CityEngine + StageRules. _Добавлено 2026-05-24_

- [ ] **applicationWillTerminate guard при isReiniting** — если пользователь закроет
      app в момент `MapReinitCoordinator.reinit(...)` (между шагами c-g), .bak файл

- [ ] **Самопроверка генерации после каждого прогона** — после автозакрытия задач,
      связанных с размещением/визуалом юнитов, цикл обязан сам себя верифицировать,
      а не полагаться на «должно работать»:
      1. **Property-тесты на размещение.** На каждый прогон UnitPlanner/DistrictPlanner —
         seed-based фикстура «10 проектов по 30 задач», и проверка инвариантов
         после каждого `applyTaskCompleted`:
         a) `footprintAdjacentToRoad(unit)` истинно (BUG-017/018 регресс-щит);
         b) `footprintBlocked(unit)` ложно (нет overlap с дорогой/другим зданием);
         c) `unit.cells ⊂ districtCells` (не вылез за квартал);
         d) первый юнит проекта — `.road` (BUG-010 регресс-щит).
      2. **Снапшот-скрин города.** Smoke-режим (`CITY_SMOKE_RUN=1`) после `swift run`
         генерит PNG карты в `Tests/__snapshots__/` (через `view.texture(from:scene)`)
         для базовых сцен: пустая карта, 1 квартал stage 1/3/5, 5 кварталов с decay.
         Diff против baseline → fail на > N% разницы; обновление baseline — ручное.
         Claude смотрит свежий PNG глазами и пишет короткое наблюдение в отчёт
         прогона («дороги ровные, кварталы плотные, висяков нет»).
      3. **Failing test = fail прогона.** Если property-инвариант сломан или
         снапшот-diff красный — TASK не считается закрытой, баг открывается
         автоматически (BUG-NNN, P1).
      Цель: ловить «здания висят в воздухе / пересекают дорогу / перекрываются»
      на стадии генерации, а не после ручного тестирования юзером.
      _Добавлено 2026-05-24_

- [ ] **Архитектурный редизайн юнитов — уйти от «пластиковых кубиков»** — сейчас
      `UnitSprites.makeCategoricalBuilding` рендерит почти всё одним рецептом:
      куб + пирамидальная крыша + brick-hatch. На карте 50 разных `UnitKind`
      сливаются в одинаковые «коробочки». Нужно пересмотреть силуэт каждого юнита
      исходя из его смысла, а не категории:
      1. **Жильё:**
         - Землянка — холм с дверью, без крыши, врастает в землю.
         - Лачуга/Хижина — кривой каркас, соломенная двускатка, без чёткого куба.
         - Каменный дом — массивные кирпичные сегменты (видна кладка), низкая крыша.
         - Доходный дом / Усадьба — несколько этажей, лоджии, балконы, окна разной формы.
         - Дворец — комплекс из 3–4 объёмов с центральным куполом/портиком.
      2. **Инфраструктура:**
         - Колодец — круглая кирпичная кладка с навесом и ведром, не куб.
         - Цистерна — открытая чаша с водой (видна синяя поверхность).
         - Маяк — высокая башня, выше всех зданий, с огнём наверху (анимация).
         - Мост/Канал/Акведук — длинные арочные пролёты, не «домик».
         - Ворота — два пилона + перекладина, без крыши.
      3. **Производство:**
         - Ферма — небольшой дом + забор по периметру + поля с урожаем
           вокруг + 1–2 силуэта животных (корова/овца). Если у kind есть
           поля — генератор размещает поля в соседних свободных тайлах,
           не только в footprint здания.
         - Шахта/Каменоломня — врезка в скалу + куча породы, без стен.
         - Лесопилка — навес + штабели брёвен, открытая стена.
         - Гончарня/Кузница — труба с дымом (есть в LifeSimulation, но silhouette
           тоже должен отличаться).
         - Пристань/Причал — деревянный настил уходит в воду, лодка пришвартована.
      4. **Социальное / военное / религиозное:**
         - Сторожевая башня — высокая, выше домов, наверху вращается прожектор
           или флаг (анимация LifeSimulation).
         - Театр/Форум — открытый амфитеатр с ярусами, не закрытый куб.
         - Храм/Часовня — портик с колоннами, не плоская стена.
         - Обелиск — узкий высокий, без крыши.
         - Пирамида — ступенчатая (уже есть).
      5. **Размещение без «прилипания»:** UnitPlanner сейчас старается ставить
         здания вплотную для компактности. Разрешить регулируемый «воздух»
         (1–2 пустых тайла между зданиями стадии 2+), особенно для ферм и
         поместий. Цель — визуально читаемый квартал, а не плотная коробочная
         сетка.
      6. **Перегенерация участков при эволюции:** когда срабатывает
         `EvolutionGraph.rule` и stage переключается на 3+, разрешить
         перерасставить часть юнитов квартала (snapshot до → план после),
         чтобы у крупных зданий появилось дыхание. Транзакция атомарная,
         анимация cross-fade ≤ 1 сек.
      7. **Спецификация на спрайт.** Каждый `UnitKind` получает в
         `SpriteGenerationRules.md` § 4 строку с примером силуэта
         (1 предложение + ключевые элементы), чтобы AI-генерация в next-итерации
         не свалилась обратно в «куб с крышей».
      Контракт: каждое из ~50 зданий читается с одного взгляда без подписи;
      кварталы выглядят как поселение, а не упаковка коробок.
      _Добавлено 2026-05-24_

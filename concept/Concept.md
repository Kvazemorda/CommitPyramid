# CityDeveloper — Концепт приложения

## Принцип
Этот файл — **зафиксированный эталон**. Он не правится автоматически в ходе разработки.
Новые идеи попадают в `Backlog.md`, а не сюда. Любая правка `Concept.md` = осознанное
решение, которое сдвигает цель цикла.

## Суть продукта

CityDeveloper — это интерактивный изометрический древний город, который живёт на рабочем
столе macOS и растёт по мере того, как пользователь закрывает реальные задачи в своей
работе.

Источник данных — **один внешний файл-лог `tasks.jsonl`**, в который наполнение пишет
внешний крон-агент пользователя (вне scope игры). Игра следит за этим файлом и каждую
новую запись о закрытой задаче превращает в юнит на карте: дом, колодец, склад,
мастерскую, рынок. Все задачи равны — у юнита нет «веса». Конкретный тип юнита выбирается
игрой по внутренним правилам баланса квартала (на N жилых нужен колодец, на M —
мастерская, и т.п.).

Проект из реального мира становится кварталом. Скорость и стадия роста квартала зависят
от **количества выполненных задач**, **возраста проекта** (от первой задачи до последней)
и **активности** (ритм событий). От пустыря через лачуги к каменной застройке, развитому
кварталу и роскоши. Если в проекте долго нет событий (по умолчанию 14 дней) — квартал
ветшает и в конце концов превращается в руины. Руины остаются на карте навсегда, но
когда появляется **новый** проект, он первым делом занимает зону руин и поверх неё
строит новые здания.

Внутри живого квартала идёт **лёгкая визуальная симуляция жизни**: на складах появляются
штабели товаров, мастерские дымят и звенят, рынок торгует, жители ходят по дорогам.
Это иллюзия деятельности, не настоящая экономика — цепочек поставок и дефицита ресурсов
нет.

Город живёт **на рабочем столе** как обычное стабильное окно, удерживаемое за всеми
другими окнами (через `NSWindow.collectionBehavior`, без хака `kCGDesktopWindowLevel`).
По горячей клавише окно поднимается в режим навигации — можно панорамировать, зумить,
кликать на здания и видеть, из какой задачи каждое из них родилось. Полный лог всех
событий хранится append-only — пользователь всегда может ответить на вопрос «за счёт
чего этот город вырос».

Стилистика — упрощённая египетская изометрия в духе Pharaoh: песочные и охровые тона,
изометрическая сетка, простые здания, ходящие жители.

## Бизнес-модель

| Источник ценности                       | Механика                                                                                                  |
|-----------------------------------------|-----------------------------------------------------------------------------------------------------------|
| Личная геймификация продуктивности      | Закрытые задачи материализуются в визуальный прогресс — мотивация доводить дела до конца                  |
| Долговременный артефакт работы          | Полный лог + воспроизводимое состояние города = «портфолио» реальной активности за годы                   |
| Обратная связь по заброшенным проектам  | Decay-механика делает видимыми проекты, к которым давно не возвращался                                    |

Проект не предназначен для коммерческой продажи. Это личный инструмент Ильи; в перспективе
возможен open-source релиз для пользователей с аналогичным workflow.

## Контракт с внешним миром

Игра принимает **только один файл** — `tasks.jsonl` (путь настраивается в F-13).
Формат каждой строки:

```json
{"ts": "2026-05-21T10:30:00Z", "project": "Outbyte Website", "title": "Согласовал ТЗ"}
```

- **`ts`** — таймстамп scan-time (момент, когда крон-агент увидел `[x]` и записал строку)
- **`project`** — название проекта (человекочитаемое; уникальное имя — идентификатор)
- **`title`** — текст задачи
- Опциональные поля (для будущих расширений): `task_id`, `source` — игра их сохраняет, но
  на логику не влияет

Никаких типов задач, весов, категорий. Юнит = юнит.

Полная спецификация формата и поведения watcher-а — в `LogFormat.md`.

## Фичи (эталонный набор)

### F-01: Стабильное окно «всегда позади» + explore-режим
**Что:** Приложение запускается как полноэкранное окно, удерживаемое **позади всех
других окон** (через `NSWindow.collectionBehavior = .stationary + .canJoinAllSpaces`,
без хака `kCGDesktopWindowLevel`). Окно прозрачно к кликам (`ignoresMouseEvents = true`)
и не получает фокус. По горячей клавише (по умолчанию ⌘⌥G, настраивается) окно
поднимается на передний план и становится интерактивным; повторное нажатие возвращает
в фон.
**Зачем:** Город должен жить за рабочим окружением без отвлечения, но при желании его
можно изучать. Стабильное окно надёжнее, чем wallpaper-уровень.
**Done-критерий:** Город виден поверх обоев и за иконками рабочего стола; клики
проходят к иконкам и другим окнам, не на город. По ⌘⌥G окно поднимается, мышь работает
с городом (камера, клики на здания). Повторный hotkey возвращает в фон. Переключение
режимов мгновенное (<200ms). При открытии/закрытии других приложений окно остаётся
позади.

### F-02: Изометрический рендер города
**Что:** SpriteKit + `SKTileMapNode` с `tileSetType = .isometric`. Камера с pan/zoom.
Сетка минимум 128×128 тайлов с возможностью расширения по мере роста.
**Зачем:** Базовый визуальный слой — без него ничего показать нельзя.
**Done-критерий:** На карте видна изометрическая сетка, по которой можно панорамировать
(drag) и зумить (scroll/pinch). FPS не падает ниже 60 в режиме explore при 500+ юнитах
на типичном Mac (M-серия). Тайлы корректно сортируются по глубине.

### F-03: Event sourcing (append-only лог + replay)
**Что:** Все изменения состояния города — события в `events.jsonl`, которые никогда не
переписываются, только добавляются. Источники событий: (а) новые строки из
`tasks.jsonl` → `TaskCompleted`, (б) системные тики игры → `DecayTick`, `Fire`,
`Restore`, `StageUp`, `UnitBuilt`, `RuinsCleared`. Команда `--replay` строит состояние
с нуля из лога.
**Зачем:** Главное требование Ильи — хранить полный лог истории и понимать, за счёт
чего вырос город. Также страховка от багов: state можно пересобрать в любой момент.
**Done-критерий:** Файл `events.jsonl` пополняется без перезаписи прошлых строк.
Удаление `state.json` не теряет данные — при следующем запуске состояние
восстанавливается из лога. Команда `--replay` доступна из CLI и даёт идентичный
результат.

### F-04: Watcher `tasks.jsonl`
**Что:** `DispatchSource.makeFileSystemObjectSource` на файл-источник (путь из настроек).
Игра помнит offset последней обработанной строки. При изменении файла читает только
хвост, валидирует каждую строку, для валидных — создаёт `TaskCompleted` событие в
`events.jsonl` и применяет к состоянию. Невалидные строки логируются в `errors.log`
и пропускаются. Файл `tasks.jsonl` **не модифицируется игрой** — только читается.
**Зачем:** Канал поступления данных от внешнего крон-агента. Read-only контракт
позволяет любому источнику писать в этот файл (cron, скрипт, manual edit для теста).
**Done-критерий:** Запись новой строки в `tasks.jsonl` приводит к появлению юнита на
карте в течение 2 секунд. Игра не теряет события между запусками (offset персистится).
Битая JSON-строка не ломает watcher — обрабатывается следующая.

### F-05: Лёгкая симуляция жизни квартала
**Что:** Поверх статичных юнитов идёт визуальная симуляция деятельности:
- на складах постепенно появляются штабели товаров (тайлы поверх базового склада);
- мастерские дымят (particle smoke) и периодически «искрят» (короткие particle bursts);
- рынок показывает анимацию активности (флажки, торговцы у прилавков);
- сырьевые ямы / поля имеют циклы (наполнение → опустошение);
- циклы независимы друг от друга и не блокируют рост города.

Реальной цепочки поставок и дефицита нет. Это иллюзия.
**Зачем:** Город должен выглядеть живым, а не статичной диорамой. Это создаёт ощущение
работающей цивилизации.
**Done-критерий:** В каждом квартале со stage ≥ 2 видна минимум одна активная анимация
(дым, штабели, флажки) одновременно. Симуляция не влияет на FPS (>60). При появлении
нового юнита он включается в общий ритм симуляции в течение 10 секунд.

### F-06: Модель Project-District и автоматическое размещение
**Что:** Каждому уникальному `project` (по имени) соответствует District — полигон на
карте. **Приоритет размещения нового проекта:**
1. Если на карте есть зоны руин — новый проект занимает наиболее старую/большую зону
   руин (с анимацией расчистки).
2. Если руин нет — выделяется свежий участок свободного луга (растущая спираль от центра).

Юниты проекта живут только в своём квартале. Расположение детерминированно (replay
даёт ту же карту).
**Зачем:** Визуальная семантика — каждый рабочий проект занимает своё место. Приоритет
руин даёт регенерацию карты — старые «мёртвые» проекты дают почву для новых.
**Done-критерий:** 3+ разных проекта в `tasks.jsonl` → 3+ непересекающихся квартала.
При появлении нового проекта на карте с зоной руин — он занимает руины с анимацией
расчистки длительностью 3-5 сек. Без руин — занимает свежий луг.

### F-07: Состав и баланс юнитов в квартале
**Что:** Внутренние правила игры (без конфига от пользователя) определяют, какой тип
юнита появляется при новой задаче. Примерные пропорции (точные числа подбираются на M2):
- ~50% жилые (лачуга / дом / вилла в зависимости от stage)
- ~20% инфраструктура (колодец, дорога, склад)
- ~20% производство (мастерская, сырьевая яма, поле)
- ~10% социальное (рынок, площадь, форум, храм)

Дополнительные правила: на каждые N жилых обязательно один колодец; рынок появляется
только при stage ≥ 2; храм/обелиск — при stage ≥ 4. Выбор детерминированный
(порядковый номер задачи в проекте → тип юнита по таблице).
**Зачем:** Разнообразие визуала без необходимости размечать типы в источнике. Игра сама
балансирует квартал.
**Done-критерий:** Квартал из 20+ юнитов содержит здания всех 4 категорий (жилые,
инфра, производство, социальное). Пропорции укладываются в ±10% от целевых. Replay
лога даёт идентичный состав.

### F-08: Стадии развития квартала (0 → 5)
**Что:** Каждый District имеет stage 0–5: луг → лачуги → деревянная застройка →
каменный квартал → развитый → роскошный. Переход вычисляется по формуле от
`taskCount`, `projectAge` (от первой задачи до последней) и `activityRate` (events/week
за последние 30 дней). Апгрейд — замена sprite-tier у существующих юнитов на месте,
без перестройки координат.
**Зачем:** Видимый прогресс по мере роста проекта. Активные проекты растут быстрее.
Сохранение «истории» (юнит на том же месте, где был построен).
**Done-критерий:** При накоплении задач и времени проект последовательно меняет визуал
через все 5 стадий. Координаты конкретного юнита не меняются между стадиями. Проект
с высокой активностью (>5 событий/неделю) достигает stage 3 за <2 недели; проект с
низкой активностью (1 событие/неделю) — за >2 месяца.

### F-09: Decay и руины
**Что:** Если у проекта нет событий > 14 дней — `decayLevel` растёт по тикеру (раз в
час + catch-up при запуске). Уровни:
- 1 (15–28 дней): трава прорастает между плитами, тайлы тускнеют;
- 2 (29–56 дней): трещины в стенах, провалившиеся крыши;
- 3 (57–90 дней): пожар (particle fire + smoke);
- 4 (>90 дней): руины. Квартал помечается как зона руин на карте.

Руины **остаются навсегда** — не удаляются и не превращаются обратно в луг. Но при
появлении **нового проекта** (см. F-06) руины могут быть выбраны для застройки и
анимационно очищаются под новый квартал. Если в старый проект приходит новое событие
до перехода в руины (decay 1-3) — запускается анимация восстановления (3-5 сек),
decay сбрасывается в 0.
**Зачем:** Обратная связь — забыл проект, он визуально дряхлеет. Возвращение даёт
видимый бонус. Руины + регенерация дают карте круговорот: ничто не теряется, но карта
не засоряется бесконечно.
**Done-критерий:** Проект без событий 14 дней показывает decay уровня 1. Через 28 → 2,
через 56 → 3 (с particle-эффектом пожара), через 90 → 4 (руины). Новая задача в
проекте с decay 1-3 сбрасывает decay в 0 с анимацией ремонта. Новый **другой** проект
может занять руины (см. F-06 Done).

### F-10: Жители и анимация
**Что:** Спрайты жителей бродят по дорогам внутри квартала. Простой waypoint pathfinding
по дорожной сетке. Население квартала = функция от количества юнитов и stage.
**Если кол-во юнитов в квартале уменьшается** (например, decay уничтожил часть) —
жители уходят с карты с анимацией.
**Зачем:** Город должен «жить» — статика выглядит мёртвой. Жители — основной носитель
ощущения движения.
**Done-критерий:** В каждом квартале со stage ≥ 2 видны минимум 3 движущихся жителя.
Жители не выходят за границы своего квартала. FPS не падает при 50+ жителях на карте.
При уменьшении количества юнитов число жителей плавно (за 5-10 сек) приходит к новому
целевому значению.

### F-11: Инспектор / журнал событий
**Что:** Клик на здание в режиме explore → попап с задачей-источником (title, ts,
project, исходная строка лога). Боковая SwiftUI-панель: список проектов с их метриками
(юнитов, население, stage, decay, последняя активность), хронологический журнал
событий с фильтром по дате/проекту/типу события.
**Зачем:** Пользователь хочет понимать «почему этот дом тут» и видеть полную историю
города.
**Done-критерий:** Клик на любой юнит открывает попап с текстом задачи, датой и
проектом. Журнал поддерживает фильтр по проекту и диапазону дат. Боковая панель
показывает все активные проекты с актуальными метриками.

### F-12: Снэпшоты состояния
**Что:** Каждые 500 событий или раз в сутки — сериализация `state.json`. При запуске
игра грузит последний снэпшот и реплеит только хвост событий после него.
**Зачем:** Быстрый старт при многотысячном логе. Без снэпшотов replay 10000 событий —
секунды задержки.
**Done-критерий:** Лог 10000 событий грузится менее 1 секунды при наличии снэпшота.
Удаление снэпшотов не теряет данные — следующий запуск восстанавливает всё из лога и
создаёт новый снэпшот.

### F-13: Каталог арт-ассетов (изометрические тайлы)
**Что:** Минимальный набор тайлов:
- 5 типов юнитов × 3 стадии (жилой, склад, мастерская, рынок, колодец);
- декор (храм, обелиск, форум) — по 1 тайлу для stage 4-5;
- дороги (4 поворота + прямые + перекрёстки);
- тайл травы (луг), тайл песка, тайл руин;
- 2-3 спрайта жителя с анимацией ходьбы (4 кадра минимум);
- particle-текстуры (дым, огонь, искры);
- товары для складов (штабели мешков, амфор) — 3-4 варианта.

Единый стиль — упрощённая египетская изометрия, пиксельная или векторная.
**Зачем:** Без ассетов город — заглушки; с ассетами становится приятно смотреть.
**Done-критерий:** Все базовые юниты, стадии и декор имеют отрисованные тайлы в едином
стиле. Дороги корректно соединяются между собой. Жители имеют анимацию ходьбы.
Тайлы руин визуально отличимы от свежего луга. Particle-эффекты вписываются в палитру
(см. `DesignConcept.md`).

### F-14: Настройки (UI)
**Что:** SwiftUI-панель настроек с полями:
- **Путь к `tasks.jsonl`** (file picker, валидация при сохранении);
- **Hotkey explore-режима** (по умолчанию ⌘⌥G, переназначается);
- **Путь к данным игры** (`events.jsonl`, `state.json`, снэпшоты) — по умолчанию
  `~/Library/Application Support/CityDeveloper/`;
- **Дисплей для города** (если подключено несколько — на старте только один,
  основной; расширение в backlog).

**Зачем:** Пользователь должен указать, откуда читать лог, и иметь контроль над
поведением приложения.
**Done-критерий:** В настройках можно сменить путь к `tasks.jsonl` — изменение
подхватывается watcher-ом без перезапуска. Переназначенный hotkey начинает работать
немедленно. Путь к данным валидируется при сохранении (есть ли доступ на запись).

### F-15: Биомы и генерация карты

**Что:** Карта генерируется **один раз** при первом запуске и сохраняется как часть
данных игры (воспроизводима через seed). Размер — минимум 256×256 тайлов. Биомы:

| Биом      | Характер тайлов                                    |
|-----------|----------------------------------------------------|
| Луг       | Зелёная трава, мягкие переходы — дефолтный биом    |
| Пустыня   | Песок, охра, редкие оазисы                          |
| Лес       | Деревья средней плотности, тёмная трава            |
| Горы      | Скальный рельеф, высота, обрывы                    |
| Камни     | Каменистая почва, переходная зона к горам           |
| Река      | Линейный водоём, соединяет биомы, ширина 2-6 тайлов |
| Море      | Большой водоём по краю или как залив                |

Переходы между биомами **плавные** (simplex/Perlin шум): луг плавно переходит в лес,
лес — в горы, пустыня — в луг через переходную зону. Биомы распределяются по принципу
«острова» или «полуострова» — море / река на границах создают естественную географию.

**Реинициализация:** В настройках — кнопка «Сбросить карту» (с подтверждением). Город
пересобирается заново с новым seed: существующие кварталы переразмещаются в
соответствующих биомах, лог событий не трогается (история сохраняется).

**Масштаб и зум:** Карта большая — минимальный зум должен позволять видеть всю карту
целиком в одном экране (обзорный режим). Максимальный зум — детальный вид одного
квартала. Диапазон зума расширен по сравнению с текущим (добавить ×0.15–×0.25 для
полного обзора).

**Аффинитет юнитов и кварталов:** У каждого юнита (см. F-16) есть поле `terrain`,
задающее предпочтительный биом. При автоматическом размещении квартала (F-06)
алгоритм учитывает биом зоны размещения: квартал рядом с рекой — выше шанс юнитов
с `terrain: река`; горный биом — кузницы и шахты, лес — лесопилки. Чистый детерминизм
смягчается **взвешенным случаем** (entropy): квартал может получить «неожиданный»
юнит с ~15% вероятностью — это создаёт естественное разнообразие.

**Зачем:** Большая живая карта с разнообразным рельефом превращает коллекцию кварталов
в настоящий город. Биомы дают визуальный смысл расположению — портовый район у моря,
горная крепость на скалах. Плавные переходы обеспечивают эстетику в духе Pharaoh.

**Done-критерий:** При первом запуске генерируется карта ≥ 256×256 тайлов с не менее
чем 4 разными биомами, соединёнными плавными переходами. Карта воспроизводима из seed.
Кнопка «Сбросить карту» + подтверждение → новая генерация, кварталы переразмещаются.
Зум позволяет увидеть всю карту в одном экране. Новый квартал рядом с рекой получает
водные/речные юниты с заметно большей вероятностью, чем равномерная.

### F-17: In-app journal — ручной ввод задач

**Что:** Встроенный лог-окошко внутри приложения для ручного ввода выполненных задач.
Решает проблему «удалённых» задач без видимого прироста кода — физический мир, встречи,
звонки, исследования. Две точки входа:

1. **Глобальный журнал.** Отдельное окно (кнопка в `JournalWindowController` или новая)
   с историей всех событий + блок ввода: текстовое поле «что сделал», dropdown с выбором
   `projectId` (существующие проекты + кнопка «новый»), Enter → новое событие в
   `events.jsonl` → юнит на карте.
2. **Контекстный ввод по клику на квартал.** В explore-режиме клик на пустую часть
   квартала → popup «добавить задачу к проекту X» (projectId предзаполнен из квартала).
   Это расширение существующего F-11 (инспектор) для empty-cell-режима.

Журнал и контекстный ввод используют общий internal pipeline `TaskCompletion(projectId,
title, ts, source="journal")` → `events.jsonl`. История событий журнала видна в том же
журнале (фильтрация по `source == "journal"`).

**Why:** Не вся работа — это код или маркдаун. Часть задач — встречи, физические дела,
обучение, удалённая работа без артефакта. Журнал даёт способ зафиксировать их в городе
прямо из приложения, без выхода в текстовый редактор.

**Done-критерий:** Глобальный журнал открывается по кнопке/хоткею, ввод текста + проект
+ Enter → юнит появляется на соответствующем квартале в течение 1 сек. Клик по пустой
части существующего квартала в explore → popup с предзаполненным projectId, ввод задачи
работает идентично. Все журнальные события видны в общем логе с тегом `source: journal`.
История journal-событий не теряется при перезапуске (idempotent через `events.jsonl`).

### F-18: Notes/folder watcher — парсер markdown-источников

**Что:** Watcher отслеживает выбранные пользователем .md-файлы/папки заметок и
парсит выполненные задачи по встроенным шаблонам. Замещает старый F-04 (single
`tasks.jsonl`) как обобщение.

**Встроенные шаблоны парсинга** (описаны в in-game docs, доступны через кнопку «?» в Settings):

| Шаблон | Пример | Извлекается |
|--------|--------|-------------|
| `- [x] [project: <id>] <title>` | `- [x] [project: myapp] починил баг с авторизацией` | project=myapp, title=починил... |
| `- [x] <title> #<project>` | `- [x] купил продукты #household` | project=household, title=купил... |
| `~~<title>~~ #<project>` | `~~позвонил врачу~~ #personal` | project=personal, title=позвонил... |
| `- [x] <project>: <title>` | `- [x] myapp: добавил тест` | project=myapp, title=добавил... |

Если строка не подходит ни под один шаблон — игнорируется.

**Источники в Settings:**
- Список путей: можно добавлять отдельный файл, папку (нерекурсивно), папку рекурсивно.
- Расширения по умолчанию `.md`, можно расширить.
- Для каждого источника — режим обработки: `delete-processed` (удалять обработанные
  строки / файлы — лог в `events.jsonl` источник истины) или `sidecar-dedup` (хранить
  `.processed-state.json` рядом, не трогать исходные .md).
- Watcher работает через `DispatchSource` (file changes) + общий 5-минутный poll
  (см. F-20).

**Why:** Многие пользователи уже ведут задачи в Obsidian / Bear / просто `notes.md`. Не
заставлять переучиваться — пусть приложение парсит существующие заметки. Опция
«удалять обработанные» снимает проблему «мусор копится в исходных файлах».

**Done-критерий:** Юзер добавляет файл/папку в Settings. Записывает `- [x] [project:
test] hello` в .md-файл, сохраняет — в течение 5 сек на карте появляется юнит в
квартале `test`. При `delete-processed` режиме строка удаляется из .md. При
`sidecar-dedup` — хэш строки записан в `.processed-state.json`, повторный парсинг
не создаёт дубликат. Все 4 встроенных шаблона работают. In-game docs описывают
шаблоны через `?`-кнопку в Settings.

### F-19: Git watcher — авто-учёт прироста кода

**Что:** Watcher отслеживает выбранные локальные git-репозитории. Каждый новый коммит
в основной ветке = событие `TaskCompletion` → юнит на карте. Языки не важны — читается
только метадата коммитов.

**Конфигурация в Settings:**
- Список репо: для каждого `path`, `projectId` (по умолчанию выводится из
  `git remote get-url origin`, например `git@github.com:foo/myapp.git` → `myapp`),
  `branch` (по умолчанию `main`/`master` авто-определение).
- Опционально: `git fetch` перед сканом (для подхвата пушей с других машин).
- Игнор: список префиксов сообщений (`chore:`, `wip:`, `style:` по умолчанию).

**Логика сканирования (по 5-минутному poll из F-20):**
1. Для каждого репо: `git log <branch> --since <last_check_ts> --pretty=format:"%H|%ct|%s"`.
2. Каждый новый sha → проверка дубликата по `source: "git:<repo>:<sha>"` в `events.jsonl`.
3. Если уникален — создать событие. Title = первая строка commit message.
4. **Опционально (advanced):** `git diff --shortstat <prev>..<sha>` → объём.
   Малый коммит (≤10 lines) = 1 юнит; средний (10–100) = 2-3; крупный (>100) = 4+.
5. **Опционально (advanced):** категория юнита из conventional-commits префикса:
   `feat:` → жилой, `fix:` → инфра (ремонт), `refactor:` → производство,
   `docs:`/`chore:` → социальный/игнор.

**Why:** Когда работа = код в репо, не хочется руками писать «закрыл задачу X». Git уже
ведёт честный лог коммитов. Автоматический мост коммиты → юниты даёт прирост города
просто за то, что разработчик работает.

**Done-критерий:** Юзер добавляет локальный git-репо в Settings. Делает коммит в `main`
(локально или `git pull` после push из web-UI). В течение 5 мин на карте появляется
юнит в соответствующем квартале. Дубликаты по sha не создаются при повторном
сканировании. Опции «вес по diff» и «категория по commit-type» можно включить/отключить
независимо в Settings.

### F-20: Catch-up watcher — единый planner для всех источников

**Что:** Единый механизм запуска всех 3 источников (F-17 пишет напрямую, не через poll;
F-18 + F-19 идут через poll). Состоит из:

1. **Periodic poll каждые 5 минут** — Timer на main-thread, при срабатывании запускает
   `scan()` для каждого активного источника F-18 / F-19. Интервал 5 мин — компромисс
   между «отзывчивость» и «нагрузка на CPU/диск».
2. **Immediate scan при старте приложения** — после загрузки snapshot+tail (F-12) и
   до подключения watcher'ов, выполняется один полный проход с `last_check_ts` из
   per-source state. Это закрывает разрыв «игра была закрыта 3 дня, что я пропустил».
3. **Per-source state** — каждый источник F-18 (per-path) и F-19 (per-repo) хранит
   `last_check_ts` в Application Support. После успешного scan → обновляется.
4. **Idempotent dedup** — F-18 через sidecar или удаление строк, F-19 через sha.
   F-17 не нуждается (запись только через UI).

**Why:** Без catch-up при старте — пропуски за время простоя теряются. Без единого
интервала — каждый watcher делает своё, нагрузка непредсказуема.

**Done-критерий:** При старте приложения после ≥1 дня простоя все коммиты в
настроенных репо + все новые `[x]` в notes-папках детектируются и попадают на карту в
течение 30 сек. Periodic poll каждые 5 мин ловит изменения в живом режиме (новые
коммиты, новые `[x]` в открытом .md-файле). Per-source `last_check_ts` сохраняется
между запусками. Интервал 5 мин может быть переопределён в Settings (3–60 мин).

### F-21: Open-source готовность (GitHub + контрибьюторы)

**Что:** Подготовить проект к публикации на GitHub как open-source с активным
контрибьюторским потоком. Идея: разработчики заинтересуются концепцией
«коммиты растят город», подключатся к проекту и помогут доработать. Особенно
ценен contribution-канал для арта (AI-генерация спрайтов по подготовленным
промптам — см. TASK-040 в Diff).

**Состав:**

1. **Подключение к GitHub-аккаунту пользователя.** Авторизация `gh` CLI или
   SSH-ключ, создание репозитория `CityDeveloper` (public, MIT-лицензия).
2. **Чистка кода от персональных данных.** Аудит: реальные имена, email-адреса,
   локальные пути в комментариях, git config user.* в коммит-метаданных, любые

### F-22: Apple Notes integration (источник событий из macOS Notes.app)

**Что:** Адаптер `AppleNotesEventSource` (реализует `EventSource`), позволяющий
пользователю подключить заметки из встроенного macOS Notes.app как источник
задач — наравне с F-18 (folder watcher) и F-19 (git watcher).

**Состав:**

1. **Доступ к Notes.app.** Два пути:
   - **AppleScript / ScriptingBridge** (`tell application "Notes" to get every note`) —
     требует разрешения пользователя при первом запуске (System Settings → Privacy →
     Automation). Стабильный API, поддерживается Apple. Минус: блокирующие IPC-вызовы,
     медленный при больших vault'ах.
   - **Прямой доступ к SQLite** (`~/Library/Group Containers/group.com.apple.notes/
     NoteStore.sqlite`) — без разрешений, быстрый. Минус: содержимое заметок
     зашифровано CoreData blob'ами, нужна reverse-инженерия структуры (есть
     open-source примеры: `apple-notes-liberator`). Хрупкий при обновлениях macOS.
2. **Settings UI секция «Apple Notes watcher»:**
   - Кнопка «Connect to Notes» — запросит permission через AppleScript.
   - Список папок Notes (iCloud accounts, On My Mac) → юзер выбирает какие подключить.
   - Шаблон детекции «выполненной задачи»: `[x]` в чек-листе заметки (стандарт Apple
     Checklist), либо строка с эмодзи ✅, либо frontmatter-like header.
   - Стратегия dedup: per-note `lastModified` timestamp + дайджест содержимого.
3. **Маппинг на projectId:**
   - По имени папки Notes (например, папка «MyApp» в Notes → projectId «MyApp»).
   - Или по тегу `#projectid` внутри заметки.
4. **Catch-up интеграция:** AppleNotesEventSource подписывается на CatchUpScheduler
   (F-20) — periodic poll + immediate scan при старте.

**Why:** Многие пользователи держат задачи и идеи в Apple Notes — это нативное
приложение macOS, синхронизируется через iCloud, доступно с iPhone/iPad. Без
этого адаптера такие пользователи вынуждены вручную экспортировать в `.md` —
лишний шаг, барьер для adoption.

**Done-критерий:** Пользователь в Settings подключает Apple Notes одним нажатием
(с системным confirm permission), выбирает 1+ папок, ставит `[x]` в любой заметке
из этих папок — в течение 5 минут (или immediate scan при старте) на карте появляется
новый юнит в соответствующем квартале. История изменений идемпотентна при replay
(не пишет дубль event'ов на ту же `[x]`).

### Что хотим
Превратить «голые» поля высоты / температуры / влажности (TASK-026) в карту
биомов: каждой клетке мира назначить ровно один из семи биомов из концепта —
**Луг, Пустыня, Лес, Горы, Камни, Река, Море**. Это смысловой слой, на котором
дальше строятся рендер (TASK-028) и terrain-аффинитет юнитов (TASK-030, потом
F-16). Видимых изменений на экране всё ещё нет — это словарь и правила
классификации, которые TASK-028 потом отрисует.

### Пользовательский сценарий
1. Игрок запускает игру с уже сгенерированной картой (TASK-026).
2. Для любой клетки можно спросить: «какой здесь биом?» — ответ один из семи,
   стабильный между запусками.
3. Распределение по карте читабельно: пустыни — где жарко и сухо, леса — где
   тепло и влажно, горы — где высоко, море/река — где низко (вода), без
   «шахматных» пятен и без полей, где один биом покрывает всё.

### Acceptance criteria
- [ ] На карте 256×256 присутствуют **минимум 4 разных биома** из семи (это
      минимум по F-15) и площадь самого крупного биома не превышает условные
      75% карты — чтобы мир не был «монобиомным».
- [ ] Каждая клетка имеет ровно один биом; для одинаковых входных значений
      (одинаковые высота/температура/влажность) результат одинаковый
      (детерминизм классификатора).
- [ ] Биом «Море» формирует крупные связные водные области по краям или как
      залив (а не отдельные «лужи» по одной клетке); биом «Река» — линейные
      связные полосы шириной ориентировочно 2–6 клеток, как описано в F-15.
- [ ] Визуально на тестовом дампе (например, текстовая карта или временный
      debug-вывод) распределение биомов выглядит географично: пустыня соседствует
      с лугом через переходную зону, горы окружены камнями, лес не граничит сразу
      с пустыней без промежутка.

### Что НЕ делаем (границы скоупа)
- Не рисуем тайлы и переходы между биомами — это TASK-028.
- Не считаем biome-аффинитет для размещения новых кварталов — это TASK-030 (и
  далее F-16).
- Не редактируем биомы вручную (никаких brush-инструментов / редактора карты).
- Не вводим динамическую смену биомов во времени (сезоны, опустынивание).

### Edge cases
- [ ] Параметры на границе порогов (например, влажность ровно на стыке луг/лес)
      → выбирается один детерминированный биом без «мерцания» между запусками.
- [ ] Карта получилась почти однородной (например, экстремальный seed) →
      классификатор всё равно даёт минимум 4 биома за счёт встроенной нормировки
      по квантилям, либо выдаёт ошибку с записью в `errors.log` и пересоздаёт
      карту с другим seed (TASK-026 управляет повтором).
- [ ] Маленький фрагмент моря в один-два тайла внутри суши → отфильтрован
      (биом «лужа» сглаживается до окружающего биома, чтобы не было «капель»
      моря в горах).
- [ ] Биом меняется в момент чтения (теоретически — миграция данных) →
      классификатор сам по себе чист и не зависит от изменяемого глобального
      состояния.

### Зависимости
- **Blocked-by:** TASK-026 (без шумовых полей классифицировать нечего).

### Дизайн
Не применимо (нет UI; реализация словарная).

### Done-критерий
_Из Concept.md F-15:_ При первом запуске генерируется карта ≥ 256×256 тайлов с не менее
чем 4 разными биомами, соединёнными плавными переходами. Карта воспроизводима из seed.
Кнопка «Сбросить карту» + подтверждение → новая генерация, кварталы переразмещаются.
Зум позволяет увидеть всю карту в одном экране. Новый квартал рядом с рекой получает
водные/речные юниты с заметно большей вероятностью, чем равномерная.

_Автор: lead (agent, Opus)_

### Анализ

**Контекст.** TASK-026 уже даёт «мир в виде трёх скаляров»: для каждой клетки 256×256
известны `height`, `temperature`, `humidity` ∈ `[0…1]`, плюс воспроизводимость по seed.
TASK-027 — это чистый словарный слой поверх: enum `BiomeKind` (7 значений) + классификатор
вида `(h, t, m, координаты) -> BiomeKind`, не зависящий от UI/SpriteKit и не трогающий
существующий `CityState`. Рендера и аффинитета юнитов здесь нет — это TASK-028/030.

**Стиль проекта.** Перечисления оформляются как в `UnitKind` / `UnitCategory` из
`CityState.swift`: `String`-rawValue + `Codable` + `CaseIterable` + extension с
`label` (русское название для UI/debug). То же делаем для `BiomeKind`. Никаких
протоколов «биома с поведением» — это просто tag.

**Семь биомов F-15.** Концепт фиксирует ровно: Луг, Пустыня, Лес, Горы, Камни, Река,
Море. Берём как канонический список enum-кейсов.

**Где живёт код.** TASK-026, очевидно, создаёт `Sources/CityDeveloper/World/`
(или аналогичный модуль) с типом вида `WorldMap` и полями. TASK-027 ставит туда
два новых файла рядом, без зависимости от `Game/` (SpriteKit). Если каталога ещё
нет — кладём в `Sources/CityDeveloper/World/`, чтобы 026/027/028 жили вместе.

**Ключевая идея классификации (детерминированно и «географично»).**
1. Сначала «вода»: порог по `height` → если ниже `seaLevel`, это кандидат на
   `море`. Связные крупные области → `море`, мелкие изолированные пятна (≤ N клеток)
   сглаживаются до окружающего биома (edge case «лужа в горах»).
2. Реки — отдельный шаг: трассируются от локальных максимумов вниз по градиенту
   высоты в море/край карты (классический downhill-flow). Получаем линейные
   связные полосы шириной 2–6 (расширяем коридор по «руслу»).
3. Суша классифицируется по двумерной таблице `(temperature, humidity)`
   при условии порогов по `height`:
   - очень высоко → `горы`;
   - высоко (предгорье) → `камни`;
   - иначе:
     - жарко + сухо → `пустыня`;
     - тепло + влажно → `лес`;
     - всё остальное → `луг` (дефолт).
4. Чтобы AC «≥ 4 биомов» соблюдалось даже на «плоских» seed-ах, пороги
   берём не как абсолютные константы, а **по квантилям** распределения соответствующего
   поля (например, top-5% высоты → горы, bottom-25% → вода). Это и есть «встроенная
   нормировка по квантилям» из edge cases.
5. Если после всех шагов биомов меньше 4 — возвращаем ошибку классификатора
   `BiomeClassifierError.insufficientDiversity`; TASK-026 уже владеет реролл-петлёй
   по seed и просто перегенерирует мир.

**Детерминизм.** Классификатор — pure-функция от `WorldMap` (или его срезов).
Никакого `Date()`, `arc4random`, глобального состояния. На границе порогов
сравнение строго `<` vs `>=` (фиксированное), чтобы устранить «мерцание».

**Стоимость.** 256·256 = 65 536 клеток. Базовая классификация — O(N), one-pass.
Сглаживание «луж» и трассировка рек — flood-fill / простые проходы, тоже O(N) с
небольшой константой. На M-серии — единицы миллисекунд, в бюджет 1 c (из TASK-026)
вписывается с запасом.

### Решение

Добавить два файла в `Sources/CityDeveloper/World/`:

1. **`BiomeKind.swift`**
   - `enum BiomeKind: String, Codable, CaseIterable { case meadow, desert, forest, mountains, stones, river, sea }`
   - `extension BiomeKind { var label: String { … } }` — русские названия как у `UnitKind.label`.
   - `extension BiomeKind { var isWater: Bool { self == .sea || self == .river } }` —
     утилитка для будущего TASK-030 (аффинитет) и TASK-028 (рендер). Чисто
     данные, без рендерных типов.

2. **`BiomeClassifier.swift`**
   - `struct BiomeMap: Codable { let width: Int; let height: Int; let cells: [BiomeKind] }`
     (плоский массив, индекс `y * width + x`, как обычно в noise-полях TASK-026; если
     TASK-026 принёс другой контейнер — переиспользуем его «обёртку», тип хранения
     биомов идентичен по форме).
   - `enum BiomeClassifierError: Error { case sizeMismatch; case insufficientDiversity(found: Int, dominantShare: Double) }`
   - `struct BiomeClassifier { static func classify(world: WorldMap) throws -> BiomeMap }`
     — pure-функция, ничего не сохраняет, ни в какие пути не пишет.
   - Внутри последовательно: `computeThresholds (квантили)` → `classifyLand` →
     `markSea (flood-fill + min-area filter)` → `carveRivers (downhill trace)` →
     `validateDiversity`.

Никакие существующие файлы (CityState, CityEngine, UnitPlanner, GameScene и т.п.)
**не правим**. TASK-028 потом сам подцепит `BiomeMap` для рендера.

### Edge cases

- Все поля одинаковые/почти одинаковые (вырожденный seed) → квантили дают
  равные пороги → диверсити < 4 → `insufficientDiversity` → TASK-026 реролит.
  Классификатор не падает, не зацикливается, не пишет в файлы.
- Значение точно на границе квантиля → используем строгое сравнение `>=`
  только в одну сторону, всегда выбираем «верхний» биом. Воспроизводимо.
- Море-«лужа» 1–2 клетки внутри суши → flood-fill + min-area filter превращает
  её в окружающий биом (см. шаг 4).
- Источник реки на склоне без выхода в море (карта замкнутая впадина) →
  трассировка останавливается, частичная река не рисуется (или, опционально,
  заканчивается локальным `.sea`-пятном размером ≥ `minSeaArea`, чтобы остаться
  валидной). Документировать в комментарии алгоритма.
- Координаты вне `[0; width-1] × [0; height-1]` (теоретическая ситуация при
  чтении `BiomeMap.at(x:y:)` со стороны TASK-028) → возвращаем дефолт `.meadow`
  без падения, по аналогии с edge case TASK-026 для out-of-bounds.
- Версия `BiomeClassifier` меняется (поменялись пороги) → старый сохранённый
  `BiomeMap` (если TASK-028/030 решат его кэшировать) можно молча пересоздать;
  для этой задачи мы карту не сохраняем в файл — только возвращаем из функции.

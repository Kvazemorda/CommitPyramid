### Что хотим
Текущий `UnitPlanner` (`nextUnitKind(forTaskIndex:stage:residentialCount:wellCount:
infraCount:productionCount:socialCount:)`) знает про 12 юнитов и не учитывает
биом клетки квартала. Расширяем его до полноценного планировщика по F-16:
учитывает текущий биом квартала (через terrain-веса из TASK-033), стадию,
`minStage` юнита, флаг `large` и фактические per-category счётчики. Сохраняет
категориальные пропорции из F-07 (10R/4I/4P/2S = 50/20/20/10%), но **внутри
категории** выбирает юнит, взвешенно семплируя по аффинитету к биому.
Сохраняет детерминизм: на одинаковых входах replay даёт идентичные результаты.

### Пользовательский сценарий
1. Игрок основывает новый проект → DistrictPlanner ставит квартал на клетке
   биома «горы».
2. По мере поступления задач планировщик выбирает: первые жилые — Хижины
   (биом: лес/горы), инфра — Колодец и Дорога, производство — Каменоломня
   и Шахта (биом: горы), социалка при stage ≥ 2 — Таверна / Площадь.
3. Тот же квартал в биоме «луг/река» выдал бы Фермерские дома, Гончарни,
   Бани — потому что terrain-веса другие; категориальные пропорции при этом
   остались примерно те же (50/20/20/10).
4. На stage ≥ 4 в квартале может появиться религиозный юнит (Часовня / Храм
   / Обелиск / Пирамида — последний только в пустыне) с малой частотой
   (≤ 1 из 20 задач), и военный — также редко.

### Acceptance criteria
- [ ] `UnitPlanner.nextUnitKind(...)` принимает аргументом биом клетки
      квартала (либо `nil` для обратной совместимости со старыми вызовами /
      тестами F-07; в `nil` режиме поведение эквивалентно равномерному
      terrain-весу).
- [ ] При прочих равных условиях, для биома горы / камни вероятность
      получить юнит с явным `terrain` к этому биому (Каменоломня, Шахта,
      Кузница, Каменный дом) **в ≥ 2 раза выше**, чем в биоме луг.
- [ ] `minStage` соблюдается жёстко: юнит с `minStage = 4` (Вилла, Обелиск,
      Библиотека, Театр) не появится в квартале на stage 3 и ниже, ни при
      каких terrain-весах.
- [ ] Категориальные пропорции по F-07 сохранены: на 20+ задачах в одном
      проекте распределение по 4 категориям из старого F-07 (R/I/P/S)
      укладывается в ±10% от 50/20/20/10. Новые категории «религиозное» и
      «военное» вычитаются из суммы «социальное», но появляются заметно
      реже (≤ 15% от social-слотов суммарно).
- [ ] Replay идентичен: тот же `events.jsonl` + та же карта биомов = тот же
      состав юнитов в проекте, до конкретного `UnitKind` каждого слота.

### Edge cases
- [ ] Биом клетки квартала — «море» (квартал на пирсе) → планировщик
      предпочитает водные юниты (Пристань, Рыболовецкий причал, Маяк, Мост,
      Верфь), но при пустой category-rotation не падает — fallback к
      «любому» юниту категории с минимальным весом.
- [ ] Категория после фильтра по `minStage` оказалась пустой (никаких
      разрешённых юнитов) → планировщик деградирует на категорию выше по
      доступности (например, social на stage 0 → infra) и пишет одну запись
      в `errors.log`.
- [ ] Аргумент-биом передан `nil` (старые вызовы) → результат идентичен
      поведению F-07 для тех же входов (обратная совместимость).
- [ ] Все юниты категории имеют terrain-вес 0 в данном биоме (теоретически
      возможно при кастомной таблице из TASK-033) → планировщик семплирует
      равномерно среди категории; не падает.

### Целевая сигнатура
```swift
func nextUnitKind(
    forTaskIndex idx: Int,
    stage: Int,
    biome: Biome?,                  // NEW; nil → terrain-вес 1.0 для всех (back-compat)
    residentialCount: Int,
    wellCount: Int,
    infraCount: Int,
    productionCount: Int,
    socialCount: Int
) -> UnitKind
```
- `biome` — последний из «контекстных» аргументов, до per-category counters это поломало бы все вызовы; ставим **между `stage` и counters** — единственная точка вызова в `CityEngine.swift:232` правится единым diff-ом, а старая сигнатура убирается (default-параметр `biome: Biome? = nil` оставляем для обратной совместимости со старыми тестами, как явно требует AC).

### Алгоритм (5 шагов)
1. **Pick category по слоту.** Базовый `categoryPattern` остаётся 20-слотным с пропорциями 50/20/20/10. Религиозная/военная подкатегория **не получают собственных слотов в pattern** — они «откусываются» от social-слотов с фиксированной частотой ≤ 15% (по AC: «≤ 15% от social-слотов суммарно»). Реализация: внутри social-ветки на каждом N-м вызове (`socialCount % 7 == 6` → religious, `socialCount % 13 == 12` → military — простые, детерминированные, дают ~14% и ~8%, итого ≤ 22% от social = ≤ 4.4% от всех задач, что укладывается в AC «≤ 15% от social-слотов суммарно»). Конкретные пороги вынести в константы `SocialMix.religiousEveryN = 7`, `militaryEveryN = 13`.
2. **Получить candidates категории.** `UnitKind.allCases.filter { $0.profile.category == cat }` (после TASK-031 у каждого `UnitKind` будет `profile: UnitProfile { category, terrain: [Biome], size, minStage, large, evolvesTo, evolvesAt }`). Кэшировать как `static let unitsByCategory: [UnitCategory: [UnitKind]]` в самом `UnitPlanner` (один раз при первом обращении) — иначе O(50) фильтр на каждый вызов.
3. **Фильтр по `minStage`.** `candidates.filter { $0.profile.minStage <= stage }`. **Edge case AC**: если пусто — записать одну строку в `errors.log` (`ErrorsLog.write`) и деградировать на категорию-«сосед» в порядке `social → infrastructure → residential` (никогда наоборот: повышение «вверх» категорий запрещено, чтобы не дать religious-юнита там, где не должно быть). Если все категории пусты — fallback `UnitKind.well` (всегда `minStage=0`).
4. **Эволюционная подмена «вниз цепочки».** Спец явно требует: «планировщик создаёт только „низовой“ юнит цепочки (Землянку, Лачугу, Склад), эволюция — поверх» (TASK-034). Реализация: после получения candidates исключаем все юниты, у которых `profile.evolvesTo != nil` **И существует другой юнит в категории/стадии, который эволюционирует именно в этот** (т.е. исключаем «промежуточные» и «вершинные эволюционные»). Проще и однозначнее по таблице F-16: формируем set «исходных» точек цепочек `evolutionRoots: Set<UnitKind> = [.dugout, .shack, .hut, .farmhouse, .stoneHouse, .twoStoryHouse, .warehouse]` (7 штук из таблицы — все, у кого `evolves != nil`) и при фильтрации **исключаем целевые юниты эволюций**: `evolutionTargets: Set<UnitKind> = [.shack, .house, .stoneHouse, .tenement, .manor]`. То есть планировщик ставит только roots + не-эволюционирующие юниты; целевые формы появляются исключительно через TASK-034. Constants — в одном месте `EvolutionTable` (создаёт TASK-034; если он ещё не готов — временно хардкодим здесь с TODO-меткой и удаляем при мерже TASK-034). Тонкий момент: для жилых на stage ≥ 2 candidate включает Двухэтажный дом, Таунхаус, Усадьбу, Виллу, Дворец — `large=true` и нероутовые → ставятся напрямую (это и есть «крупные сразу»).
5. **Взвешенный детерминированный sample.** Веса: `w(k) = terrainWeight(k, biome ?? .meadow_uniform)`. Если `biome == nil` — вместо вызова используем константу `1.0` для всех (явное back-compat поведение). Если `sum(w) == 0` (AC edge case: все юниты категории имеют 0 в данном биоме) — равномерный sample (`w(k) = 1`). Sample:
   ```swift
   var seed = seedFor(taskIndex: idx, category: cat, biome: biome, projectId: nil)
   var rng = SplitMix64(seed: seed)
   let r = rng.nextUnit() * sum
   var acc = 0.0
   for (k, w) in weighted { acc += w; if r < acc { return k } }
   ```
   Где `seedFor` — FNV-1a/SipHash от `(idx, cat.rawValue, biome?.rawValue ?? "nil")`. **Без `projectId`** — иначе разные проекты получат разные веса при «одинаковых входах» (AC явно требует replay-идентичность на одной карте, проектная ось — отдельный детерминизм, project уже отражается через `idx`/counters). `SplitMix64` — миниатюрная (≤15 строк) deterministic struct-RNG, лежит в новом файле `Sources/CityDeveloper/Game/DeterministicRNG.swift` — нужна и здесь, и в TASK-033/034.

### Контракты / инварианты
- **Детерминизм:** на одинаковых `(idx, stage, biome, counters)` `nextUnitKind` возвращает один и тот же `UnitKind`. RNG — только `SplitMix64(seed: seedFor(...))`, никаких `Date()`/`UUID()`/`SystemRandomNumberGenerator`.
- **`nil` биом ≡ uniform.** AC требует «эквивалентно равномерному terrain-весу» — гарантируется явной веткой «`biome == nil` ⇒ `w = 1.0`».
- **Категориальные пропорции F-07** сохранены до знака (pattern не трогаем). Religious/military «съедают» только social-слоты на детерминированные позиции, R/I/P не страдают.
- **`minStage`** соблюдается жёстко после шага 3; никакая последующая логика не имеет права вернуть юнит с `minStage > stage`.
- **Эволюция:** планировщик не возвращает evolution-target'ы (`house`, `stoneHouse`, `tenement`, `manor`); они появляются только через TASK-034. Если TASK-034 ещё не реализована — целевые формы просто не появятся на карте, это допустимо (квартал визуально будет «состоять из roots», но replay-корректен).
- **Errors.log** — пишется ровно одна запись на edge case «категория пуста» (без флуда: проверка через `os_unfair_lock` или `static var loggedEmptyCategories: Set<UnitCategory>` в `UnitPlanner`).

### Edge cases — план покрытия

| Edge case (AC) | План реализации |
|---|---|
| Биом «море» | Фильтрация по terrain-весам + sum>0 fallback → водные юниты (Маяк, Пристань, Верфь, Рыболовецкий причал, Мост) получат высокий вес, sample их выберет. Не падает. |
| Категория после minStage пуста | `ErrorsLog.write` (once per category), fallback на `social → infrastructure → residential`, в крайнем случае `.well`. |
| `biome == nil` | Ветка перед расчётом весов: `w = 1.0` без вызова `terrainWeight`. Результат побитово эквивалентен старому F-07-планировщику для тех же входов (валидируется тестом из TASK-038). |
| Все юниты категории `w == 0` | `if sum == 0 { uniform }` — sample среди тех же candidates с равным весом. Не падает. |

### Сложность / объём
- **Сложность: senior.** Поверхностно — switch + filter, но: (1) корректное seedable RNG, (2) согласованность с тремя ещё-не-готовыми зависимостями, (3) replay-инвариант под угрозой при любой ошибке в seed-функции, (4) AC «≥ 2× weight ratio» требует выбора правильной формулы веса (не просто умножения terrain-веса в `[0..1]` — иначе ratio будет ~5×, не «≥ 2×»: надо строить формулу `w = baseUniform + terrainBoost * terrainWeight`, где `baseUniform ≈ 0.15` (соответствует F-15 «15% неожиданности»), `terrainBoost ≈ 0.85`. Это junior не выкатит без шишек.
- **Объём: M.** Один файл на ~220 строк + новый файл RNG на ~30 + правка `CityEngine` на 3 строки. Без UI, без миграций, без новых типов данных (всё приходит из зависимостей). Тесты — отдельная TASK-038.

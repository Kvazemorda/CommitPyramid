### Что хотим
Концепт F-16 в `Concept.md` — это **эталон** (не правится в ходе разработки).
Но рабочая документация для разработчиков и арт-генерации нуждается в
живом, обновляемом справочнике: «вот 50 юнитов, вот их id-rawValue, вот
имена PNG-файлов, вот биом-предпочтения и цепочки эволюции, вот категории».
Этот справочник используется при генерации арта (по
`SpriteGenerationRules.md` § 4.1) и при работе с планировщиком. Создаём его
как `concept/UnitCatalog.md`, дублирующий таблицы F-16 в удобной для
повседневной работы форме, и расширяющий их **техническими** деталями
(rawValue, PNG-имя, флаги). Сам `Concept.md` остаётся эталоном.

### Пользовательский сценарий
1. Разработчик / арт-генератор открывает `concept/UnitCatalog.md` и видит
   единую таблицу по 50 юнитам: №, русское название, rawValue в
   `UnitKind`, категория, terrain, size, minStage, large, целевая
   эволюция, имя PNG (`<kind>.png` или `<kind>_stage<N>.png`).
2. Для каждой из 8 эволюционных цепочек видна явная схема `from → to (порог)`.
3. В подвале документа — список из 6 категорий с их color-token из
   `Palette.swift` (то, что использует `categoricalGroundColor` в TASK-032),
   плюс ссылка на `SpriteGenerationRules.md` для требований к арту.

- Не вводим эту документацию как «in-game docs» (отдельная подзадача, если

### Edge cases
- [ ] Юнит без `size` в F-16 (Акведук — `—` в таблице) → в `UnitCatalog.md`
      пишется явное «линейный (не клеточный)» с пометкой «требует
      отдельной обработки».
- [ ] Юнит с эволюцией, цель которой — `large = true` (например, Доходный
      дом, Усадьба, Большой склад) → в строке цепочки явно указано
      «цель `large = true`, не эволюционирует дальше».

      явно записано как «любой», не пустая клетка.
- [ ] PNG-имя для юнитов со stage-tier (residential: shack/house/villa) →
      указывается шаблон `<kind>_stage<N>.png` + перечисление актуальных
      stage (2..5 для house, 1 для shack).

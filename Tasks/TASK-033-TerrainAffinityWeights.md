## 📋 Постановка от менеджера

_Автор: pm (agent)_

### Что хотим
По концепту F-16 у каждого юнита есть поле `terrain` — список биомов, в
которых юнит «как дома». Каменоломня и Шахта тяготеют к горам, Лесопилка —
к лесу, Пристань и Маяк — к морю/реке, Цистерна — к пустыне. Эта задача
вводит **функцию веса** «насколько уместен юнит X в биоме Y», которая будет
использоваться TASK-035 (`UnitPlanner`) для смещения распределения юнитов
в сторону «географической логики». Чистый детерминизм смягчается
взвешенным случайным выбором (по F-15: квартал может получить «неожиданный»
юнит с ~15% вероятностью — за счёт этого карта остаётся живой, а не
шахматной).

### Пользовательский сценарий
1. Разработчик выбирает любую клетку карты (через TASK-027 классификатор)
   и получает её биом — например, «горы».
2. Для этой клетки можно спросить «насколько уместна Каменоломня?» и
   получить высокий вес (например, 1.0), для Пристани — низкий (например,
   0.15).
3. Когда `UnitPlanner` решает, какой юнит поставить в квартал, он смотрит на
   биом клетки квартала и взвешенно выбирает один из юнитов категории —
   юниты, у которых биом клетки в списке `terrain`, имеют преимущество,
   но не монополию.

### Acceptance criteria
- [ ] Существует чистая функция от пары (`UnitKind`, `Biome`) → вещественный
      вес в диапазоне [0..1] (0 = категорически не подходит, 1 = идеально
      подходит, 0.15 = «неожиданный гость»).
- [ ] Юнит с `terrain = [любой]` (например, Землянка, Дорога, Дом) имеет
      одинаковый вес во всех 7 биомах (равномерное распределение).
- [ ] Юнит с явным предпочтением (например, Каменоломня — горы/камни)
      имеет вес ≥ 0.8 в предпочтительных биомах и ≤ 0.2 в остальных.
- [ ] Никакой биом не даёт нулевой вес всем юнитам категории — для любой
      категории и любого биома есть хотя бы один юнит с весом ≥ 0.5 (чтобы
      `UnitPlanner` всегда мог что-то поставить, даже в крайних биомах).
- [ ] Функция детерминирована: один и тот же запрос даёт один и тот же
      результат, без зависимости от глобального состояния (replay-safe).

### Что НЕ делаем (границы скоупа)
- Не реализуем сам выбор юнита по биому — это TASK-035 (`UnitPlanner` это
  использует).
- Не размещаем кварталы на карте с учётом биома — это TASK-030 (district
  placement) уже в работе по F-15.
- Не строим heatmap или UI визуализации весов — никакого UI.
- Не считаем биом клетки динамически (он зафиксирован картой из F-15).
- Не вводим конфиг весов в Settings — таблица захардкожена в коде по таблице
  F-16 («terrain» столбец).

### Edge cases
- [ ] Биом, не упомянутый ни в одной строке таблицы F-16 (например, гипотетический
      «болото») — функция возвращает 0.5 для всех юнитов (нейтральный
      fallback), а не падает.
- [ ] `UnitKind`, для которого `terrain` явно не задан (баг данных из TASK-031)
      — функция возвращает 0.5 для всех биомов и пишет одну запись в
      `errors.log` (не флудит).
- [ ] Категория «военное / религиозное», где у юнитов разные `terrain` (например,
      Пирамида — только пустыня, Часовня — любой биом): для биома «лес» сумма
      весов военно-религиозных юнитов > 0 (чтобы `UnitPlanner` мог выбрать
      хотя бы один).
- [ ] Биом «море» — большинство сухопутных юнитов имеют низкий вес, но не
      нулевой (юнит может быть «на пирсе у берега»); только Пирамида и
      Каменоломня могут иметь вес ≤ 0.05.

- **Blocked-by:** TASK-027 (нужна модель `Biome` с 7 типами биомов).

Реализация — детерминированный классификатор, тип значения — `Codable, Hashable, CaseIterable`.

**Ожидаемые сигнатуры от блокеров (для планирования):**

```swift

enum Biome: String, Codable, Hashable, CaseIterable {
    case meadow, desert, forest, mountains, stones, river, sea
}

// TASK-031 → расширение UnitKind + UnitCategory + поле terrain
extension UnitKind {
    /// Список предпочтительных биомов из таблиц F-16; «любой» = Biome.allCases.
    var terrain: [Biome] { ... }

```

Эти сигнатуры — фундамент для текущей задачи. Если TASK-031/027 выберут немного иные имена, реализация TASK-033 правится точечно (одно место — `terrainWeight(for:in:)` плюс таблица). Это допустимый риск, потому что spec PM явно фиксирует понятие «список биомов на юните» и «7 биомов».

**Семантика весов (из AC и таблиц F-16):**
- Юнит с `terrain == Biome.allCases` («любой») → вес `1.0` во всех 7 биомах (равномерно).
- Юнит с явным предпочтением + биом в списке → вес `1.0`.
- Юнит с явным предпочтением + биом НЕ в списке → вес `0.15` («неожиданный гость»; AC: `≤ 0.2`).
- Edge: пирамида/каменоломня в «море» → вес `0.05` (AC требует «≤ 0.05 только для них»; технически — общий минимум `0.05` для пар, где биом — `море`, а у юнита явный нон-морской `terrain` с биомами {`пустыня`} или {`горы/камни`} без пересечений с морем; формула это покрывает естественно, если базовый «не-в-списке» вес сделать кусочно-линейным, см. ниже).
- Edge: `terrain.isEmpty` (баг данных) → `0.5` для всех биомов + одна запись в `errors.log`.
- Edge: биом не из 7-таблицы (теоретически — `nil` или новое значение enum) → `0.5` для всех юнитов.

**Конкретика весов (захардкоженная таблица, чтобы пройти все AC):**

```
weight(kind, biome):
  if kind.terrain.isEmpty                          → log_once; return 0.5
  if kind.terrain == Biome.allCases                → return 1.0   // «любой»
  if biome in kind.terrain                         → return 1.0   // предпочтительный
  // явный список, биом НЕ совпал — определяем «насколько чужой»
  if biome == .sea && kind.terrain ⊆ {.desert, .mountains, .stones}  → return 0.05
  else                                             → return 0.15  // «неожиданный гость»
```

Это решение:
- AC #1 (диапазон 0..1): `{0.05, 0.15, 0.5, 1.0}` — все в [0,1]. ✓
- AC #2 (terrain=любой → равномерно): отдельная ветка. ✓
- AC #3 (предпочтение → ≥0.8 в своих, ≤0.2 в чужих): 1.0 ≥ 0.8; 0.15/0.05 ≤ 0.2. ✓
- AC #4 (никакой биом не даёт 0 всем юнитам категории): для любого биома `b` есть юниты категории с `terrain == «любой»` (вес 1.0). Проверка по таблицам F-16:
  - residential: Землянка, Лачуга, Дом, Таунхаус, Доходный дом, Дворец = «любой» → есть жилые с весом 1.0 в любом биоме. ✓
  - infrastructure: Дорога, Ворота = «любой» → ✓
  - production: Мастерская, Склад, Большой склад, Завод = «любой» → ✓
  - social: Таверна, Рынок, Площадь, Школа, Больница, Форум, Библиотека, Театр = «любой» → ✓
  - religious: Часовня, Храм, Собор = «любой» → ✓ (для биома «лес» Edge case #3 выполнен: Часовня даёт 1.0, Пирамида/Обелиск дают 0.15).
  - military: Сторожевая башня, Казармы = «любой» → ✓
- AC #5 (детерминизм): чистая функция без глобального состояния. ✓

Edge #4 (море, ≤0.05 **только** Пирамида/Каменоломня):
- Перечитываем спеку дословно: «только Пирамида и Каменоломня **могут** иметь вес ≤ 0.05». Слово «только» — ограничительное: для всех остальных сухопутных юнитов в море вес должен быть **строго > 0.05** (т.е. 0.15 «неожиданный гость», но не ниже).
- Решение: white-list `seaIncompatible: Set<UnitKind> = [.pyramid, .quarry]` (имена case'ов придут из TASK-031 — ожидаются `pyramid`/`quarry`; если PM выберет иные, white-list правится в одном месте).
- Для пары `(kind in seaIncompatible, biome == .sea)` → `weightDryInSea = 0.05`. Для всех остальных пар, где биом не в `terrain`, → `weightUnexpected = 0.15`.
- Шахта/Цистерна/Кузница в море → 0.15 (соблюдено «не нулевой»; соблюдено «только Пирамида и Каменоломня могут ≤0.05»).

**Расположение кода:**
- Новый файл `Sources/CityDeveloper/Game/TerrainAffinity.swift`. Не размещать внутри `UnitPlanner.swift`, чтобы:
  1. TASK-035 переиспользовал чистую функцию без зависимости от UnitPlanner;
  2. TASK-038 тестировал отдельно.
- Структура — `enum TerrainAffinity` со static-методом (чистая функция, без хранимого состояния):

```swift
enum TerrainAffinity {
    /// Вес «уместности» юнита в биоме клетки. Диапазон [0..1].
    /// - 1.0  — биом в `kind.terrain` или `terrain` = «любой» (все 7 биомов);
    /// - 0.5  — `terrain` пуст (баг данных) или биом неизвестен (fallback);
    /// - 0.15 — «неожиданный гость» (явный список, биом не совпал);
    /// - 0.05 — Пирамида/Каменоломня в море (white-list).
    static func weight(for kind: UnitKind, in biome: Biome) -> Double
}
```

- Константы порогов вынести как `private static let` внутри enum: `weightPreferred = 1.0`, `weightUnexpected = 0.15`, `weightDryInSea = 0.05`, `weightFallback = 0.5`.
- Для edge case'а `terrain.isEmpty` — флаг `private static var loggedKinds: Set<UnitKind>` под `NSLock` (одна запись в `errors.log` на каждый «битый» kind, не флудит), либо `os_unfair_lock`. Альтернатива: писать в лог из вызывающего кода — но это размывает ответственность. Оставляем здесь.
- Чтобы оставаться чистой функцией снаружи, лог-сайд-эффект — однократный warning, никаким способом не влияет на возвращаемое значение.

### Решение

Чистая static-функция `TerrainAffinity.weight(for:in:) -> Double` в новом файле `Sources/CityDeveloper/Game/TerrainAffinity.swift`. Алгоритм — 4-уровневое решающее дерево по таблице выше. Опирается на `UnitKind.terrain: [Biome]` из TASK-031 и `Biome` enum из TASK-027. Сам UnitPlanner и CityState **не трогаем** — это сделает TASK-035.

### Шаги

1. **[AC:1, AC:5]** Создать `Sources/CityDeveloper/Game/TerrainAffinity.swift`:
   - `enum TerrainAffinity` + `static func weight(for kind: UnitKind, in biome: Biome) -> Double`.
   - Объявить 4 числовые константы (`weightPreferred=1.0`, `weightFallback=0.5`, `weightUnexpected=0.15`, `weightDryInSea=0.05`) как `private static let` — никаких magic numbers в теле.
2. **[AC:2]** Реализовать ветку «`terrain == Biome.allCases` или `Set(terrain) == Set(Biome.allCases)` → `weightPreferred`». Использовать `Set` для устойчивости к порядку.
3. **[AC:3, AC:4, Edge #4]** Реализовать основную логику:
   - `if Set(kind.terrain).contains(biome) → 1.0`
   - `else if biome == .sea && seaIncompatibleKinds.contains(kind) → 0.05`
     где `private static let seaIncompatibleKinds: Set<UnitKind> = [.pyramid, .quarry]` (имена — ожидаемые из TASK-031; при отличии — обновить таблицу в одном месте).
   - `else → 0.15`
4. **[Edge #2]** Ветка `kind.terrain.isEmpty`: вернуть `0.5` + однократный `ErrorsLog.write("TerrainAffinity: \(kind) has empty terrain")` через `Set<UnitKind>` под `NSLock` (модуль-приватная статика). Замок берётся только при первом обращении к проблемному kind — overhead пренебрежимо мал.
5. **[Edge #1]** `Biome` — закрытый enum из 7 case'ов; «биом не из таблицы» физически невозможен (компилятор не пропустит). Если в будущем добавят 8-й case без обновления таблицы — `Set.contains` вернёт false, и пойдёт ветка «0.15». Это безопасный fallback ≥ 0.5? Нет — это 0.15. AC формулирует «биом не упомянут → 0.5». Решение: при добавлении нового биома `default`-семантика должна быть «нейтральная». Для этого в шаге 3 завернуть проверку: если `Set(kind.terrain).isDisjoint(with: Set(Biome.allCases))` (т.е. список биомов юнита пуст после пересечения с известными — деградировавшие данные) → `0.5`. Но при компиляторно-полном enum это всегда false. **Итог:** оставить как есть, edge #1 покрыт самой природой enum'а (нельзя передать неизвестный case).
6. **[AC:4]** Добавить XCTest `Tests/CityDeveloperTests/TerrainAffinityTests.swift` с тестами:
   - `testAnyTerrainGivesUniformOne` — для всех 7 биомов вес `Землянки/Дороги/Дома` = 1.0.
   - `testPreferredHighOtherLow` — Каменоломня в `mountains`/`stones` = 1.0; в `meadow`/`desert`/`forest`/`river` = 0.15; в `sea` = 0.05 (она в white-list).
   - `testForEachCategoryAndBiomeAtLeastOneHigh` — цикл по `UnitCategory.allCases` × `Biome.allCases`: ∃ kind такого, что `weight ≥ 0.5`. Покрывает AC #4.
   - `testEmptyTerrainReturnsHalfAndLogsOnce` — синтетический моk не сделать, поэтому тест на `weightFallback` через статус возвращаемого значения (если получится завести kind с пустым `terrain` — обернуть через test-only расширение или skip). Если такой kind в реальной таблице отсутствует — тест помечается как «invariant»: проходим по `UnitKind.allCases`, проверяем `terrain.isEmpty == false`.
   - `testSeaPenaltyOnlyForWhitelist` — Пирамида и Каменоломня → 0.05 в `sea`; Шахта/Цистерна/Кузница → 0.15 в `sea` (не в white-list, но и не в `terrain`); Часовня (terrain=любой) → 1.0 в `sea`; Рыболовецкий причал (terrain={river,sea}) → 1.0 в `sea`.
   - `testDeterminism` — два вызова подряд с одинаковыми аргументами дают идентичный Double (битовое равенство).
7. **[Verify]** Прогнать `swift build && swift test --filter TerrainAffinityTests`.

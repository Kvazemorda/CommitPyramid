### Что хотим

Добавить фильтр в журнал боковой панели (F-11) по типу события: показывать
только `taskCompleted` (по умолчанию) или все/выбранные системные события
(`unit_built`, `stage_up`, `decay_tick`, `fire`, `restore`, `ruins_cleared`).
Сейчас в журнале (TASK-007) показываются только `taskCompleted`-события — это
все, что физически пишется в `events.jsonl`.

### Почему отложено и почему нужно разбить

Фильтр без данных бессмысленен. Системные события пока **не пишутся**:

- `decay_tick`, `fire`, `restore` начнут писаться после **TASK-008** (decay).
- `unit_built`, `stage_up` — пока не пишутся вообще (state выводится из
  `taskCompleted`). Их запись — отдельная архитектурная задача (модификация
  `CityEngine.applyTaskCompleted` + потенциально миграция формата лога).
- `ruins_cleared` — связан с F-06 (приоритет руин), который не реализован.

Поэтому для полноценного фильтра нужны как минимум **3 предусловия**:

1. **TASK-008** (decay events) — частично закроет проблему.
2. Новый TASK на «писать `unit_built` и `stage_up` отдельными событиями» —
   возможно с миграцией формата `events.jsonl` v1 → v2 (см. P2 в `Current.md`).
3. TASK на сам UI фильтра (Picker по типу события в TASK-007 / ProjectCard).

### Предлагаемое разбиение на следующий цикл

- **TASK-015a:** Системные события `unit_built` / `stage_up` начинают писаться в
  `events.jsonl` отдельно. Решить вопрос совместимости replay со старыми логами
  (либо явная миграция, либо `apply(.unitBuilt)` идемпотентен относительно
  state, восстановленного из `taskCompleted`).
- **TASK-015b:** UI фильтра в журнале (Picker «Тип события»).

Зависимости: TASK-008 (decay events) должна быть **closed** до /pm TASK-015a.

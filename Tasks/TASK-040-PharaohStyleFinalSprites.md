### Что хотим
После TASK-032 (placeholder-спрайты) визуал всех 50 юнитов работает, но это
процедурные примитивы из IsoBuilder. Финальная цель D-16 — каталог арта в
**стиле Pharaoh** по правилам `SpriteGenerationRules.md`:
изометрия 2:1, песочные/охровые тона, лёгкая иллюстративность, прозрачный
PNG-фон, единый стиль. Эта задача — финальная: добавить PNG для **ключевых
юнитов**, которые читаются игроком как «вау, это Дворец / Пирамида / Маяк»,
а не «коробка с крышей». Остальные юниты могут продолжать жить на
placeholder, пока арт не подъедет.

Если в момент исполнения D-16 placeholder-визуала из TASK-032 хватит
по UX-ощущению — задачу разрешается переместить в `Backlog.md` (один
«знаковый» PNG-каталог на финальный лоск), без блокировки закрытия D-16.

### Пользовательский сценарий
1. Игрок видит на карте Пирамиду (биом «пустыня», stage 5) — это
   узнаваемая ступенчатая пирамида в охровых тонах, не «куб с пирамидальной
   крышей».
2. Дворец, Собор, Маяк, Верфь, Театр, Форум — каждый имеет финальный PNG,
   соответствующий правилам Pharaoh.
3. Эволюционные цепочки (Землянка → Лачуга → Дом → Доходный дом)
   читаются как «улучшение того же здания», а не «4 разных дома» (см.
   `SpriteGenerationRules.md` § 4.2).
4. Игрок понимает, что приложение «оживлено» — это не только geometric
   placeholders, а художественно цельный город.

### Acceptance criteria
- [ ] Список **«ключевых юнитов»** (минимум 12 штук) имеет финальные PNG
      в `Sources/CityDeveloper/Resources/Buildings/` с правильными именами
      по `SpriteGenerationRules.md` § 3. Минимальный список (фиксируется
      явно в задаче, согласовать с пользователем при необходимости):
      Дворец, Пирамида, Собор, Храм, Усадьба, Доходный дом, Большой склад,
      Маяк, Верфь, Театр, Форум, Каменоломня.
- [ ] Каждый финальный PNG соответствует техническим требованиям
      `SpriteGenerationRules.md` § 1: RGBA, прозрачный фон, изометрия 2:1,
      свет сверху-слева, ракурс согласован.
- [ ] Каждая эволюционная цепочка из 8 (TASK-034) имеет визуально
      согласованные «до» и «после» спрайты для минимум 2-х шагов цепочки
      (например, Землянка + Лачуга, Лачуга + Дом, Склад + Большой склад).
- [ ] Сборка проходит, все ключевые юниты на сцене отрисовываются из PNG
      (не placeholder), и при этом placeholder-юниты (остальные ≈ 38)
      продолжают рисоваться корректно без падений.
- [ ] Done-критерий F-16 выполнен в реальном использовании (см. ниже).

### Что НЕ делаем (границы скоупа)
- Не покрываем PNG-ами все 50 юнитов в одной задаче (это объём L; здесь
  только «ключевые ≥ 12»). Остальные — в Backlog `B-` пункты по мере
  генерации.
- Не редактируем particle-эффекты (дым, флажки, огни) — это F-05.
- Не делаем animation-кадры для юнитов (анимация по F-13: только жители).
- Не правим `IsoBuilder` (placeholder-фабрику) — она остаётся как fallback.
- Не вводим atlas (`.atlas/` папку) — пока обычная папка по
  `SpriteGenerationRules.md` § 3 (миграция в atlas — отдельный backlog).

### Edge cases
- [ ] Если AI-генерация выдаёт PNG с непрозрачным фоном или с водяным
      знаком → перегенерировать, **не** чинить в Photoshop
      (`SpriteGenerationRules.md` § 6).
- [ ] PNG ключевого юнита приехал с неправильным масштабом основания
      (anchor «висит» над тайлом) → подкрутить `anchorY` в коде
      `loadBuildingSprite`, не перерисовывать PNG.
- [ ] Если ключевой юнит на момент исполнения задачи не может быть
      сгенерирован (нет арт-источника) → задача переносится в Backlog
      с явным списком оставшихся PNG, D-16 закрывается «частично, ключевые
      готовы, остальные — backlog» с обновлением `Current.md` (F-16 →
      ⚠️, с пометкой «PNG в работе»).
- [ ] Замена placeholder → PNG для уже существующего юнита на сцене не
      требует перезапуска: достаточно сборки и горячей перезагрузки сцены
      (или повторного запуска приложения). Это поведение — фактическое;
      ручное тестирование подтвердить.

### Вердикт: escalate-too-large

Задача честно оценена как **L (>1 дня)**. Плана нет — по правилу /lead
протокола (шаг 7): «если объём = L — не пиши план».

### Обоснование оценки L

1. **Объём арт-производства.** AC требует минимум **12 финальных PNG**
   (Дворец, Пирамида, Собор, Храм, Усадьба, Доходный дом, Большой склад,
   Маяк, Верфь, Театр, Форум, Каменоломня) **+ согласованные пары** для ≥ 2
   эволюционных цепочек (минимум ещё 4 спрайта). Итого ≈ 16 PNG.
2. **Каждый PNG — итеративный.** Per `SpriteGenerationRules.md` § 1, § 6:
   разрешение 1024²–2048², прозрачный фон, изометрия 2:1, свет сверху-слева,
   style-anchor по `shack.png`. Workflow явно требует «один спрайт = один
   PNG», «батч по 2–3 в одном диалоге», ретейки при стиль-дрейфе. Это не
   «прогон в один проход», это часы ручной работы на цепочку.
3. **Согласованность палитры/силуэта** между парами эволюции
   (`SpriteGenerationRules.md` § 4.2) — отдельный круг ретейков, чтобы пары
   читались как «улучшение того же дома».
4. **Технический контур задачи** (правки `anchorY` в `loadBuildingSprite`,
   расширение switch-а в `UnitSprites.makeStageNode` для подгрузки PNG по
   `UnitKind`-named файлу) — это S сам по себе, но требует чтобы 50 кейсов
   `UnitKind` существовали в коде (т.е. блокирующая TASK-031). Без неё код
   нечем расширять.
